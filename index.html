Written Questions

  1. Describe how your dashboard answers the questions presented. You don't have to address every question directly, but should at a high level address the main questions. (10 points)
    • For graph 1, I used a bar graph to show the top 10 games of all time or a specific year based on global sales. An interactive select was used to allow the user to change the time frame. The bars can be clicked to display the top 10 publishers for the game's genre in graph 3.
    • For graph 2, I used a map to show the top genre for each region and a pie chart to show the breakdown of a region's sales by genre. Each region is clickable to change which region the pie chart displays. Each pie slice can be clicked to show the top publishers in that genre.
    • For graph 3, I used a bar graph to show the top 10 publishers of a genre based on global sales. An interactive select was used to allow the user to change the genre. The top publisher is also bolded to display the clear top publisher.
  2. List 3 reasons why D3 was helpful and improved your visualization (6 points)
    • D3 made it very easy to implement the graphs, especially with the examples on their website that could be used as a basic template to start from. This allowed me to focus more on how to represent the data, rather than worrying too much about the implementation.
    • D3's structure also made it easy to create interactive elements and have graphs that handle the transitions and changes of different data. This made the visualizations more interesting and engaging.
    • D3's tooltip was also extremely customizable. This allowed me to improve the layout of the web page and have positions and color the way I wanted.
  3. List 3 reasons why D3 would not be the best tool for creating a visualization (6 points)
    • Even though D3 uses SVGs, there may be a better visualization tool for people who rely on screen readers. The graphs displayed by D3 are not the best way to represent data for blind people.
    • Although it is very customizable, it was not the quickest way to get to the final graphs. Since there is so much to be specified, if someone is in a time rush, it may be better to use something like Excel.
    • It is not very intuitive and the code is rather difficult to read. When multiple people are collaborating on a visualization, it may be better to use a simpler program so that everyone is on the same page on how the visualization is made.
  4. Color blind people might find this dashboard accessible. First, I made sure that the colors contrasted enough in different color blindness using the color blind tool in the coolors.co site. Also, all of my graphs do not depend explicitly on color, and for areas of some uncertainty, I have a tooltip to clear up any confusion. Also for the pie chart, I included white space to distinguish the different sections. On the other hand, blind people or people with low vision might have a more difficult time with this dashboard. First, the text is not that large, which may be hard to read. Also, although the graphs are all svgs, it would be difficult to determine the exact information the graph is portraying just from a screen reader. Along with this, the map and the pie chart uses tooltips which rely on hovering, and this is near impossible with screen readers. One thing I did include was a dropdown input since these can be used by screen readers. These are all aspects mentioned by Amanda Miller in A Comprehensive Guide to Accessible Data Visualization. From her article, one way to improve my dashboard would be to get rid of the hover tooltips and replace them with labels that are always present. These labels should be directly on the graphs, instead of using a color-reliant key.
